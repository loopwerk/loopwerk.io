# Q1 2026 in review

Normally I just stick to my yearly "year in review" articles to report on my open source contributions, but the first three months of this year have been so productive that a Q1 update seems warranted.

The reason? Well, at the beginning of January my full-time client work came to an end, freeing up a lot of time for open source work. It also means that [I am available for new projects](/hire-me/). If you'd like to add this kind of productivity to your team, [reach out to me](mailto:kevin@loopwerk.io).

Here's what I shipped in three months.

## Critical Notes
After neglecting it for too long, I returned to [Critical Notes](https://www.critical-notes.com) and implemented multiple improvements, two of them fairly significant:

1. Better user subscriptions. Instead of limiting free users to a single campaign, everyone can now join unlimited campaigns. Free users have read-only access to all but their first campaign, which is still a substantial improvement. The most common complaint I received was that people felt "forced" to pay when their group wanted to start a second campaign. This update should solve that problem for most users.
2. Data export. DMs can now export their campaign and receive a zip file containing Markdown files for all content, all uploaded images, and a JSON file with a complete raw data dump.

## Saga
I spent most of this quarter working on [Saga](https://github.com/loopwerk/Saga). In three months I published 35 [releases](https://github.com/loopwerk/Saga/releases), which is honestly too many. I have since gotten better at bundling changes together and doing fewer but larger releases, but early on I was shipping updates multiple times per week, sometimes even multiple times per day.

Here are ten of the biggest improvements:

1. Much better build times using incremental builds, a read cache, and a persistent fetch catch.
2. Full support for internationalization, including fully translatable URLs and locale-aware sitemaps.
3. Add next/previous pagination to the item rendering context. This makes it much easier to create a link to the next article for example.
4. Items can now be loaded programmatically using a new `register(metadata:fetch:writers:)` pipeline step. Because sometimes your content doesn't live on disk in markdown files, but comes from an API, or from a database, or you just want to create your items programmatically.
5. Pages can now be created programmatically as well. It was kind of awkward to just render a HTML template to disk, without a markdown file triggering this. Think of a 404 page, or a contact form. There's no `contact.md` to render, it's just purely a template. Now there's the `createPage` pipeline step just for this.
6. A new `postProcess` step, for things like HTML minification.
7. A new `hashed` function that automatically generates cache-busting hashed URLs for your static assets.
8. Full support for Swift 6. Until recently your Saga projects had to use the Swift 5 language mode, because otherwise your builds would crash. Now Saga is fully Swift 6 mode compatible, without causing new warnings or problems for existing users using the Swift 5 language mode. This took a lot of trial and error to get right, and a big chunk of the 30 releases were about Swift 6 support, improving it bit by bit. While the Swift 5 language mode is still fully supported, a Swift 6 compiler is required now.
9. Updated documentation, with 10 detailed how-to guides for things like adding search to your site, Tailwind CSS, adding HTML minification, and more.
10. And last but certainly not least: nested pipelines! When you have nested items, such as photos inside of albums, you can now naturally model this with Saga, and easily render an index of albums, a page for each album, and a page for each photo. Or think about Artists with Albums with Tracks. All of this is expressible by your Saga pipeline now, all with strongly typed metadata.

I also created a brand new [Saga CLI](https://github.com/loopwerk/saga-cli) with `saga init`, `saga dev`, and `saga build` commands, completely replacing the old `swift run watch` command.

Starting a new project is now much easier: `saga init` generates a fully scaffolded project, and the new development server is far more reliable. It detects file changes more consistently, no longer depends on an externally installed `browser-sync`, and works on Linux. The CLI can be installed via Homebrew or Mint.

## GetSaga.dev
Saga has always had solid documentation, generated with Apple's DocC compiler, and hosted on [Github Pages](https://loopwerk.github.io/Saga/documentation/saga/). While functional, it had several drawbacks: the URLs are ugly, the site is not visually appealing, and it cannot serve as a proper landing page on its own domain.

So I built a new site: [GetSaga.dev](https://getsaga.dev).

It uses the same DocC documentation files but renders them using Saga and Apple's [SymbolKit](https://github.com/swiftlang/swift-docc-symbolkit). In other words, Saga now builds and renders its own documentation, including the full API reference, which I think is pretty cool.

## New open source projects
This quarter also produced several new open source projects:

* [Moon](https://github.com/loopwerk/Moon): a server-side syntax highlighter using Prism via JavaScriptCore, fully compatible with Linux.
* [Sigil](https://github.com/loopwerk/Sigil): SymbolKit to syntax-highlighted HTML, used by GetSaga.dev.
* [Bonsai](https://github.com/loopwerk/Bonsai): HTML minification, a Swift port of html-minifier-next, with pretty good performance.
* [SwiftTailwind](https://github.com/loopwerk/SwiftTailwind): Tailwind CSS for Swift projects.
* [SagaUtils](https://github.com/loopwerk/SagaUtils): HTML transformations for Saga powered by SwiftSoup.
* [saga-cli](https://github.com/loopwerk/saga-cli): the brand new Saga command line app, installable via Homebrew and Mint.
* [SagaPathKit](https://github.com/loopwerk/SagaPathKit): a fork of PathKit, extended with new functionality and better tests. It's used now in Saga itself.

I even designed logos for most of these projects using [Affinity](https://www.affinity.studio).

A couple of my existing projects also received updates:

## Loopwerk.io
I improved the mobile navigation bar, added light mode support, and redesigned the homepage. I have also already published 18 articles in 2026.

By far the most popular one so far was my [Tim Cook article](/articles/2026/tim-cook-sold-apples-soul/), which received [quite a lot of views](https://hachyderm.io/@kevinrenskers/116012266579862557).

Articles now use server-side syntax highlighting via [Moon](https://github.com/loopwerk/Moon), removing the need for browsers to load Prism.js client-side. This noticeably improves performance on slower connections and significantly increased the PageSpeed score.

## Moving from Plausible to Umami
I migrated all my sites from [Plausible Analytics](https://plausible.io) to a self-hosted [Umami](https://umami.is) instance.

Almost immediately it became clear just how many bots are hammering my sites. I had to [take action](/articles/2026/protect-analytics/) to prevent the analytics data from becoming completely useless.
